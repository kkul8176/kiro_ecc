# 에이전트 라우팅 규칙

> 이 규칙은 모든 사용자 메시지에 대해 적용된다. `development-workflow.md`의 기능 구현 워크플로우보다 우선한다.

## 핵심 규칙

모든 에이전트(Kiro 기본 에이전트 포함)는 사용자가 요청한 내용에 대해 작업 내용을 파악한 후, **제일 먼저 해야 할 일은 등록된 에이전트 목록에서 해당 요청에 적합한 에이전트가 있는지 확인하는 것이다.**

- 적합한 역할의 에이전트가 있다면: `invokeSubAgent`를 사용하여 해당 에이전트를 호출하고 작업 내용을 전달한다.
- 적합한 에이전트가 없다면: 요청을 받은 에이전트가 직접 처리한다.

## 라우팅 판단 기준

요청 유형과 에이전트 매칭 예시:

| 요청 유형 | 적합한 에이전트 |
|-----------|----------------|
| 모호한 요구사항, 새 프로젝트/기능 요청 | prompt-generator |
| 시스템 설계, 아키텍처 결정 | architect |
| 구현 계획 수립, 작업 분해 | planner |
| UI/UX 디자인, 화면 설계 | designer |
| 코드 리뷰, 품질 검토 | code-reviewer |
| 보안 취약점 검토 | security-reviewer |
| 빌드/타입 에러 해결 | build-error-resolver |
| 테스트 작성 (TDD) | tdd-guide |
| E2E 테스트 | e2e-runner |
| 데이터베이스 관련 | database-reviewer |
| 문서 업데이트 | doc-updater |
| 리팩토링, 데드코드 정리 | refactor-cleaner |
| AWS 클라우드 아키텍처 설계 | cloud-architect |
| IaC 코드 작성, 인프라 구성, 클라우드 배포 | iac-engineer |
| Go 코드 리뷰 | go-reviewer |
| Go 빌드 에러 | go-build-resolver |
| Python 코드 리뷰 | python-reviewer |
| 단순 질문, 설명 요청, 일반 대화 | 직접 처리 (invoke 불필요) |

## 예외 사항

다음 경우에는 에이전트 라우팅을 건너뛴다:

1. **단순 질문/설명 요청**: "이게 뭐야?", "차이점 알려줘" 같은 정보 요청은 직접 응답
2. **사용자가 특정 에이전트를 지정한 경우**: 사용자가 명시적으로 에이전트를 지정하면 그대로 따름

## 릴레이 파이프라인 (에이전트 간 협업)

### 기본 원칙
서브 에이전트는 다른 서브 에이전트를 직접 invoke할 수 없다 (Kiro 플랫폼 제약). 대신 **Kiro 기본 에이전트가 릴레이 오케스트레이터** 역할을 수행한다.

```
Kiro (릴레이 오케스트레이터)
  → 에이전트 A invoke → 결과 수신
  → Kiro가 결과를 분석하고 다음 에이전트 판단
  → 에이전트 B invoke (A의 결과를 prompt에 포함) → 결과 수신
  → 반복...
```

### 릴레이 시 필수 규칙

1. **컨텍스트 전달**: 이전 에이전트의 결과를 다음 에이전트의 prompt에 핵심 내용을 요약하여 포함한다. 전체 결과를 그대로 넘기지 않고, 다음 에이전트가 필요로 하는 정보만 추출하여 전달한다.
2. **릴레이 추적**: 릴레이 파이프라인 진행 중에는 현재까지 invoke한 에이전트 목록과 각 결과 요약을 내부적으로 추적한다.
3. **판단 주체는 Kiro**: 각 에이전트의 결과를 받은 후, 다음에 어떤 에이전트를 invoke할지는 Kiro가 판단한다. 에이전트가 "다음에 X를 호출해달라"고 요청할 수 있지만, 최종 결정은 Kiro가 한다.

### 리뷰 루프 (구현 ↔ 리뷰 반복)

코드 구현 후 code-reviewer를 invoke하고, 리뷰 결과에 따라 수정 → 재리뷰가 필요한 경우:

1. **Kiro가 직접 수정하고** code-reviewer를 다시 invoke한다 (같은 에이전트 재호출 허용).
2. **리뷰 루프는 최대 5라운드**까지 허용한다.
3. 각 라운드에서 Kiro는 **심판 역할**을 수행한다:
   - code-reviewer의 지적이 모두 타당하면 → 전체 반영 후 재리뷰
   - 일부만 타당하면 → 타당한 것만 반영 후 재리뷰
   - 지적이 과도하거나 사소하면 → 현재 코드 유지, 루프 종료
4. 5라운드 후에도 미해결 이슈가 있으면 → **사용자에게 에스컬레이션** (현황 요약 + 결정 요청)

### 동일 에이전트 연속 호출 규칙

- **허용**: 리뷰 루프에서 code-reviewer 재호출 (수정 후 재리뷰)
- **허용**: build-error-resolver 재호출 (수정 후 빌드 재확인)
- **금지**: 동일 에이전트를 동일한 입력으로 연속 호출 (결과가 달라질 이유가 없음)
- **최대 연속 호출**: 같은 에이전트를 연속 5회까지만 허용. 5회 초과 시 사용자에게 에스컬레이션.

### 전체 릴레이 깊이 제한

하나의 사용자 요청에 대해 invoke하는 에이전트의 **총 횟수는 최대 10회**를 초과할 수 없다. 10회 도달 시:
- 현재까지의 진행 상황을 사용자에게 보고한다.
- 추가 진행 여부를 사용자에게 확인받는다.

### 에스컬레이션 조건 (강제 중지)

다음 조건 중 하나라도 해당하면 릴레이를 즉시 중단하고 사용자에게 보고한다:

1. **동일 내용 반복**: 에이전트가 이전 라운드와 실질적으로 동일한 결과를 반환 (진전 없음)
2. **리뷰 루프 5라운드 초과**: code-reviewer ↔ 수정 루프가 5회를 넘김
3. **전체 invoke 10회 초과**: 하나의 요청에서 에이전트 호출이 10회를 넘김
4. **에이전트 실패**: 서브 에이전트가 작업을 완료하지 못하고 에러를 반환

## 복합 요청 처리

하나의 요청에 여러 역할이 필요한 경우:

1. 요청을 분석하여 필요한 역할들을 식별
2. 가장 핵심적인 역할의 에이전트를 먼저 invoke
3. 결과를 분석하고, 후속 작업이 필요하면 다음 에이전트를 invoke (릴레이)
4. 각 에이전트의 결과를 다음 에이전트의 prompt에 컨텍스트로 포함
5. 릴레이 깊이 제한과 루프 제한을 준수

### 복합 요청 릴레이 예시

```
"이 기능 설계하고 구현해줘"
  → architect invoke (요구사항 전달) → 설계 결과 수신
  → planner invoke (설계 결과 요약 포함) → 구현 계획 수신
  → Kiro가 계획에 따라 직접 구현
  → code-reviewer invoke (구현 코드 포함) → 리뷰 결과 수신
  → [리뷰 루프: 필요시 수정 → code-reviewer 재호출, 최대 3라운드]
  → security-reviewer invoke (최종 코드 포함) → 보안 리뷰 수신
  → 완료 보고

"이 코드 리뷰하고 보안 점검도 해줘"
  → code-reviewer invoke → 리뷰 결과 수신
  → security-reviewer invoke (코드 + 리뷰 결과 포함) → 보안 리뷰 수신
  → 통합 결과 보고
```

# 산출물 규약 (Artifact Protocol)

> 이 규약은 에이전트가 작업 결과를 파일로 남길 때 따라야 하는 **공통 규칙**이다.
> 각 에이전트가 어떤 산출물을 만드는지는 해당 에이전트의 프롬프트(`.kiro/agents/*.md`)에 정의되어 있다.
> 이 steering은 "형식, 위치, 검증 방법"만 규정한다.

## 핵심 원칙

1. **텍스트 보고 < 파일 산출물**: 릴레이 파이프라인에서 다음 에이전트가 참조할 결과물은 반드시 파일로 남긴다. 채팅 응답만으로 핸드오프하지 않는다.
2. **docs/ 디렉토리 집중**: 모든 산출물 문서는 프로젝트의 `docs/` 디렉토리에 생성한다.
3. **검증 가능**: 산출물 존재 여부는 파일 시스템에서 직접 확인할 수 있어야 한다.
4. **자기 선언**: 각 에이전트는 자기 프롬프트에 "내가 만드는 산출물"을 선언한다. 중앙 카탈로그에 등록할 필요 없다.

## 산출물 파일 공통 형식

모든 산출물 마크다운 파일은 다음 헤더로 시작한다:

```yaml
---
title: {산출물 제목}
agent: {에이전트 이름}
project: {프로젝트명}
date: {YYYY-MM-DD}
type: {architecture | plan | review | security | design | deploy | cloud}
relay_step: {몇 번째 에이전트 invoke인지, 예: 3/5}
tags:
  - kiro/artifact
  - kiro/{에이전트 이름}
  - project/{프로젝트명}
related:
  - "[[관련 산출물 제목]]"
---
```

> 이 YAML frontmatter는 Obsidian 프로퍼티와 호환된다. Obsidian에 동기화 시 Graph View, Dataview, Bases에서 산출물을 구조화하여 조회할 수 있다.

### 태그 체계

| 태그 패턴 | 용도 | 예시 |
|-----------|------|------|
| `kiro/artifact` | 모든 에이전트 산출물 공통 | 필수 |
| `kiro/{agent}` | 에이전트별 분류 | `kiro/architect`, `kiro/planner` |
| `project/{name}` | 프로젝트별 분류 | `project/arm-review-system` |
| `type/{type}` | 산출물 유형별 분류 | `type/architecture`, `type/review` |

### 위키링크 규칙

- 산출물 간 참조 시 `[[산출물 제목]]` 위키링크를 사용한다.
- 릴레이 파이프라인에서 이전/다음 산출물을 `related` 프로퍼티에 링크한다.
- 프로젝트 인덱스 노트(`[[프로젝트명]]`)를 허브로 참조한다.

## 산출물 파일명 규칙

- 형식: `docs/{역할}-report.md` (예: `docs/review-report.md`, `docs/security-report.md`)
- 설계 문서: `docs/architecture.md`
- 구현 계획: `docs/implementation-plan.md`
- 코드 외 결과물(디자인, IaC 등)은 별도 폴더 허용 (예: `design/`, `cloud/`)

에이전트가 새로 추가되면, 그 에이전트의 프롬프트에 산출물 파일명을 선언하기만 하면 된다. 이 steering을 수정할 필요 없다.

## 산출물 매니페스트 (프로젝트별)

### 언제 생성하나?
새 프로젝트를 시작할 때 (planner 또는 architect가 invoke된 경우), 해당 에이전트는 `docs/artifact-manifest.md`를 생성한다.

### 어떻게 생성하나?
planner/architect는 다음을 수행한다:
1. `.kiro/agents/` 디렉토리의 에이전트 목록을 확인한다.
2. 프로젝트 성격을 분석하여 어떤 에이전트가 참여할지 판단한다.
3. 참여 에이전트의 프롬프트에서 산출물 선언을 읽는다.
4. 이를 바탕으로 매니페스트를 동적으로 구성한다.

### 매니페스트 형식

```markdown
# 산출물 매니페스트

- 프로젝트: {프로젝트명}
- 생성일: {YYYY-MM-DD}

## 산출물 목록

| 산출물 파일 | 담당 에이전트 | 필수/선택 | 상태 |
|------------|-------------|----------|------|
| docs/architecture.md | architect | 필수 | ⬜ 미생성 |
| docs/implementation-plan.md | planner | 필수 | ⬜ 미생성 |
| ... | ... | ... | ... |
```

### 매니페스트 업데이트
각 에이전트가 산출물을 생성하면, Kiro(릴레이 오케스트레이터)가 매니페스트의 상태를 업데이트한다:
- `⬜ 미생성` → `✅ 완료` (파일 존재 확인 후)

## 릴레이 핸드오프 규칙

### 다음 에이전트에게 전달할 때
Kiro가 다음 에이전트를 invoke할 때, prompt에 다음을 포함한다:

1. **참조할 산출물 파일 경로**: "설계 문서는 `docs/architecture.md`를 참조하세요"
2. **이전 에이전트의 핵심 결론 요약**: 전체 파일을 복사하지 않고 핵심만 요약
3. **이 에이전트에게 기대하는 산출물**: "작업 완료 후 `docs/{역할}-report.md`를 생성해주세요"

### 산출물 검증
릴레이 파이프라인의 마지막 단계에서, Kiro는 매니페스트의 필수 산출물이 모두 `✅ 완료` 상태인지 확인한다.
- 누락된 산출물이 있으면 해당 에이전트를 다시 invoke하여 보완을 요청한다.
- 검증은 **파일 존재 여부**로 판단한다 (에이전트의 텍스트 보고만으로 통과시키지 않는다).

## 에이전트 프롬프트에 산출물을 선언하는 방법

각 에이전트의 `.md` 프롬프트 파일에 다음 섹션을 추가한다:

```markdown
## 산출물 (Artifact)

- **파일**: `docs/{역할}-report.md`
- **필수 섹션**:
  - {섹션 1}
  - {섹션 2}
  - ...
- **릴레이 시 다음 에이전트에게 전달할 핵심 정보**: {요약}
```

이 선언이 있는 에이전트만 산출물을 생성한다. 선언이 없는 에이전트(예: build-error-resolver)는 산출물 없이 작업 결과를 채팅으로 반환한다.

## 단일 요청 (프로젝트가 아닌 경우)

"이 코드 리뷰해줘", "이 에러 고쳐줘" 같은 단일 요청에서는 매니페스트를 생성하지 않는다. 산출물 파일도 필수가 아니다. 단, 릴레이 파이프라인이 3단계 이상 진행되는 경우에는 중간 결과를 파일로 남기는 것을 권장한다.

# 클라우드 보안 기본 원칙

> 이 규칙은 클라우드 자원을 생성/변경/설계/리뷰하는 모든 작업에 적용된다.

## 핵심 원칙

1. **최소 권한 (Least Privilege)**: IAM 정책은 필요한 최소한의 권한만 부여한다. `*` 와일드카드 사용을 지양한다.
2. **암호화 기본 적용**: 저장 데이터(at rest)와 전송 데이터(in transit) 모두 암호화한다.
3. **퍼블릭 접근 금지 원칙**: 명시적으로 필요한 경우가 아니면 자원을 퍼블릭으로 노출하지 않는다. 퍼블릭 접근이 필요하면 사용자에게 확인받는다.
4. **네트워크 격리**: 퍼블릭/프라이빗 서브넷을 분리한다. DB, 캐시 등 백엔드 자원은 프라이빗 서브넷에 배치한다.

## 보안 그룹 규칙

- 인바운드: 필요한 포트만 열고, 소스를 최대한 제한한다 (0.0.0.0/0 지양).
- 아웃바운드: 기본 허용이지만, 민감한 환경에서는 필요한 대상만 허용한다.
- 설명(description) 필드를 반드시 작성한다.

## IAM 규칙

- 서비스별 전용 IAM 역할을 생성한다 (하나의 역할을 여러 서비스가 공유하지 않는다).
- 인라인 정책보다 관리형 정책을 우선 사용한다.
- `Action: "*"`, `Resource: "*"` 조합을 금지한다.
- 임시 자격 증명(STS AssumeRole)을 사용하고, 장기 액세스 키를 코드에 넣지 않는다.

## 시크릿 관리

- 시크릿(DB 비밀번호, API 키 등)은 Secrets Manager 또는 Parameter Store에 저장한다.
- 환경 변수로 주입하되, 코드/IaC 파일에 평문으로 넣지 않는다.
- 시크릿을 로그에 출력하지 않는다.

## 체크리스트

- [ ] IAM 정책이 최소 권한을 따르는가
- [ ] 보안 그룹이 필요한 포트만 열고 있는가
- [ ] DB/캐시가 프라이빗 서브넷에 있는가
- [ ] 저장 데이터 암호화가 활성화되어 있는가 (EBS, RDS, S3)
- [ ] HTTPS/TLS가 적용되어 있는가
- [ ] 시크릿이 Secrets Manager/Parameter Store에 있는가
- [ ] 퍼블릭 접근이 필요한 자원에 대해 사용자 확인을 받았는가
